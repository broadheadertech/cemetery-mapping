# Story 3.7: Admin Voids or Cancels a Contract Pre-Interment

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an **Admin**,
I want **a `voidContract` workflow that pre-interment cancels a contract (lot → available, ownership closed, contract state → cancelled) WHILE preserving all already-issued receipts and payments as immutable historical records** (FR31),
so that **operational mistakes (customer changed mind, wrong lot picked, duplicate entry) can be undone cleanly — but the cemetery's books still show what actually happened, with a separate refund workflow handling the financial reversal out-of-band** (FR24).

This story is the **broader pre-interment void workflow** that builds atop Story 3.6's `cancelContract` foundation. The difference is scope:

- **Story 3.6's `cancelContract`** is the state-machine layer: transitions the contract + lot + ownership atomically; emits three audit rows. It is the **mechanism**.
- **This story's `voidContract`** is the **operational wrapper**: it calls 3.6's `cancelContract`, then handles the additional concerns the operator cares about — the explicit "this is a void, not a state cleanup" framing in the dialog, the warning about prior payments remaining immutable, the surfaced reference to "refund workflow handled out-of-band" (Phase 1 has no in-system refund flow; Story 4.x will own that). It also adds the **pre-interment check** as a hard policy gate (FR24's explicit requirement) — even though 3.6 includes a defensive version, this story owns the customer-facing wording and the refund-record placeholder.

The story exists because the operational UX is materially different from the bare state transition: the admin needs to know "this is what's reversible, this is what isn't" before clicking confirm. The receipts the customer holds remain valid documents (per FR31 immutability); the void records that the underlying transaction was operationally reversed; refunds happen in a separate workflow.

## Acceptance Criteria

1. **AC1 — `voidContract` mutation exists and routes through `cancelContract`**: `convex/contracts.ts > voidContract({ contractId, reasonCategory, reasonText, refundExpected })` (a) calls `requireRole(["admin"])`, (b) loads contract + lot + occupant records; if any occupant has `dateOfInterment` set, throws `INVARIANT_VIOLATION` with the message "Cannot void contract — lot has been interred. Use transfer workflow if ownership needs to change.", (c) builds the formatted reason string from `reasonCategory` + `reasonText` (e.g. `"customer_change_of_mind | Customer decided to purchase a different lot section after walk-through."`), (d) calls `cancelContract(contractId, reason)` from Story 3.6 — which handles the state-machine transitions + audit emission, (e) writes one additional `voidedContracts` audit-companion row (NEW table this story creates) with: `contractId`, `voidedAt`, `voidedBy`, `reasonCategory`, `reasonText`, `refundExpected: boolean`, `priorPaymentTotalCents` (sum of non-voided payments on the contract — informational; the void does NOT touch the payments).

2. **AC2 — `voidedContracts` audit-companion table captures the void's operational metadata**: a NEW table in `convex/schema.ts` separate from the generic `auditLog` table. The companion row is the **forensic anchor**: when a refund clerk later opens the void to process the refund (Phase 2 / Epic 4), they read this row, not the three audit rows. The `voidedContracts` row contains the structured operator-supplied data (`reasonCategory` is from a closed enum); the audit log captures the raw before/after state changes. Both exist; both are queried for different forensic purposes.

3. **AC3 — Already-issued receipts and payments remain immutable** (FR31): a Vitest test posts an installment sale (down payment recorded → receipt issued with serial `OR-0000007`), then voids the contract. After void: (a) the `payments` row's `isVoided` flag is **still `false`** — the payment was real, the void doesn't retroactively pretend it wasn't, (b) the `receipts` row's `isVoided` flag is **still `false`** for the same reason, (c) the receipt's `serialFormatted` is still `OR-0000007` and the receipt PDF (when generated by Story 3.11) still bears that serial, (d) a Phase 2 refund workflow (Epic 4) reads `voidedContracts` + the contract's `priorPaymentTotalCents` to compute what to refund and issues a separate counter-receipt at that time. **In short:** voiding the contract does NOT void the receipts. Receipts are voided ONLY via Story 3.12 (`postFinancialEvent({ kind: "void_receipt" })`).

4. **AC4 — Contract detail page surfaces the void affordance and consequences**: extends Story 3.6's "Cancel" button. The button label changes based on context:
   - Contract is `active` AND lot has no interments: button label **"Void contract"** (this story's full workflow).
   - Contract is `active` AND lot HAS interments: button is disabled with tooltip "Cannot void — lot has been interred. Use the transfer workflow."
   - Contract is `fully_paid` / `cancelled` / `in_default` / `transferred`: button is hidden.
   Clicking opens the `VoidContractDialog` (NEW component) which (a) shows the structured reason form (category dropdown + free-text), (b) shows the static warning block: "Voiding this contract: lot returns to `available`, ownership ends today, no further payments can be recorded. Already-issued receipts (₱8,000.00 total from 2 payments) remain valid official documents. Refunds must be processed separately." (c) has a Confirm + Cancel; Confirm calls `voidContract` mutation.

5. **AC5 — Test-enforced: void of a contract with prior payments leaves the financial ledger intact**: tests assert that after `voidContract`, the sum of `payments.amountCents` for the contract is unchanged (every prior payment is still there, every `isVoided` is still `false`), the receipts table still has the original rows, the `receiptCounter.currentSerial` is unchanged (no new serial consumed by the void). Forensic verification path: querying `auditLog.by_entity` on the contract returns the three Story 3.6 transition rows + a fourth row from this story's `voidedContracts` companion lookup. The total audit trail is debuggable without surprise.

### Mutation (AC1, AC3)

- [ ] **Task 2: Add `voidContract` to `convex/contracts.ts`** (**UPDATE** Story 3.6's file) (AC: 1, AC: 3)
  - [ ] Export:
    ```ts
    export const voidContract = mutation({
      args: {
        contractId: v.id("contracts"),
        reasonCategory: v.union(
          v.literal("customer_change_of_mind"),
          v.literal("data_entry_error"),
          v.literal("duplicate_sale"),
          v.literal("wrong_lot_selected"),
          v.literal("payment_dispute"),
          v.literal("other"),
        ),
        reasonText: v.string(),
        refundExpected: v.boolean(),
      },
      handler: async (ctx, args) => {
        await requireRole(ctx, ["admin"]);
        if (args.reasonText.trim().length < 10) {
          throwError(ErrorCode.INVARIANT_VIOLATION, "Void reason required (min 10 chars).");
        }
        if (args.reasonText.length > 500) {
          throwError(ErrorCode.INVARIANT_VIOLATION, "Void reason too long (max 500 chars).");
        }

        const contract = await ctx.db.get(args.contractId);
        if (!contract) throwError(ErrorCode.NOT_FOUND, "Contract not found.");

        // Interment check — hard policy gate per FR24. Story 3.6's cancelContract has a defensive check;
        // this story's version is the customer-facing wording.
        const interredOccupant = await ctx.db
          .query("occupants")
          .withIndex("by_lot", q => q.eq("lotId", contract.lotId))
          .filter(q => q.neq(q.field("dateOfInterment"), undefined))
          .first();
        if (interredOccupant) {
          throwError(
            ErrorCode.INVARIANT_VIOLATION,
            "Cannot void contract — lot has been interred. Use transfer workflow if ownership needs to change.",
          );
        }

        // Snapshot prior-payment total BEFORE cancelContract runs (defensive — cancelContract doesn't touch payments,
        // but capturing the value now anchors the audit trail).
        const priorPayments = await ctx.db
          .query("payments")
          .withIndex("by_contract", q => q.eq("contractId", args.contractId))
          .filter(q => q.eq(q.field("isVoided"), false))
          .collect();
        const priorPaymentTotalCents = priorPayments.reduce((sum, p) => sum + p.amountCents, 0);

        // Compose the structured reason — preserved verbatim for forensic queries.
        const reason = `${args.reasonCategory} | ${args.reasonText.trim()}`;

        // Delegate to Story 3.6's cancelContract mutation logic — but DO NOT call it as a separate mutation
        // (Convex mutations cannot transactionally invoke each other). Instead, inline the state-transition
        // logic OR factor out an internal helper `internalCancelContract(ctx, contractId, reason)` that
        // both this mutation and cancelContract share. **Prefer the internal helper approach** — see Task 3.

        await internalCancelContract(ctx, args.contractId, reason);

        // Then write the voidedContracts companion row (this is the only NEW write in this mutation
        // beyond what cancelContract already did)
        const userId = (await getAuthedUser(ctx))._id;
        await ctx.db.insert("voidedContracts", {
          contractId: args.contractId,
          voidedAt: Date.now(),
          voidedBy: userId,
          reasonCategory: args.reasonCategory,
          reasonText: args.reasonText.trim(),
          refundExpected: args.refundExpected,
          priorPaymentTotalCents,
          refundProcessedAt: undefined,
          refundReferenceId: undefined,
        });

        return { contractId: args.contractId, priorPaymentTotalCents, refundExpected: args.refundExpected };
      },
    });
    ```
  - [ ] **Critical: do NOT touch the `payments` or `receipts` tables.** FR31 immutability. Even though the contract's outstanding balance just went to zero, you do NOT patch payments. The `paymentAllocations` rows also remain — they're historical records of how the now-voided contract's payments were allocated.

### Disaster prevention — what the dev agent must NOT do

- ❌ **Do NOT void the receipts as part of `voidContract`.** FR31 immutability. Receipts are voided via Story 3.12 (`postFinancialEvent({ kind: "void_receipt" })`) — a separate, dedicated workflow. The cemetery's BIR auditor needs the receipts to remain on the books as originally issued; the void of the underlying contract is a separate operational event.
- ❌ **Do NOT patch `payments.isVoided` from this mutation.** Same reason. Payments are only voided by Story 3.12's receipt-void path (which itself doesn't delete the payment, just flags it). Story 3.7 is purely operational.
- ❌ **Do NOT add a "Refund now" button to the dialog.** Refunds are Phase 2 / Epic 4; clicking the checkbox queues the refund in `voidedContracts`, that's all.
- ❌ **Do NOT consume a new receipt serial for the void.** No receipt is being issued. `receiptCounter.currentSerial` is unchanged. Tests verify.
- ❌ **Do NOT inline the cancellation logic** instead of calling `internalCancelContract`. Duplication produces divergence — if Story 3.6's audit-emission pattern changes, voidContract would silently miss the change.
- ❌ **Do NOT call `cancelContract` (public mutation) from inside `voidContract`.** Convex mutations cannot call other public mutations transactionally. The refactor to `internalCancelContract` (Task 3) is the architecturally correct solution.
- ❌ **Do NOT skip the interment check** with a "well, the lot transition will fail anyway" handwave. The interment check produces a clear, customer-facing error message; the lot-transition failure would produce `ILLEGAL_STATE_TRANSITION` which is the wrong UX (it suggests a state bug, not a policy reason).
- ❌ **Do NOT enable Enter as a confirm shortcut on the dialog.** UX § 1050 confidence-loop — destructive actions deserve a deliberate mouse-click or Tab+Space, not a keyboard reflex.
- ❌ **Do NOT auto-route to the lot detail page on success.** Stay on the contract detail page; let the admin see the new `Cancelled` state pill with the amber flash. Routing away erases the visual confirmation.
- ❌ **Do NOT store the reason as a single concatenated string** in `voidedContracts.reasonCategory + reasonText`. Keep them separate (closed-enum + free-text) so future filters can query by category. The `reason` passed into `internalCancelContract` is the concatenated form (for the audit log's `reason` field, which is free-text by design).
- ❌ **Do NOT make `refundExpected` derived from `priorPaymentTotalCents > 0`.** The operator should decide — sometimes a customer voids a sale before paying anything (₱0 total, no refund); sometimes a customer says "keep the down payment, I changed my mind" (₱8,000 total, no refund expected). Keep it operator-supplied.

### Common LLM-developer mistakes to prevent

- **"Voiding" vs. "cancelling" confusion:** in this codebase, **cancel** is the state-machine transition (Story 3.6); **void** is the operational wrapper (this story). The UX uses "Void contract" because that's the operator's mental model (FR24 wording). Internal code can use "cancel" for the state transition. Be consistent: UI = void; state-machine = cancel.
- **Misinterpreting FR31:** "immutable after issuance" means **the financial documents (payments, receipts) are immutable**. The CONTRACT's state can change (it's literally the point of FR23). Don't extend the immutability rule to the contract document — that would prevent the very void this story enables.
- **Refund creep:** an LLM developer reading "refund workflow handled separately" might want to wire up a refund mutation "while I'm here." DO NOT. Phase 2 owns refunds. This story stops at queuing.
- **Audit row triple-counting:** Task 3's refactor emits three audit rows via `internalCancelContract`. This story does NOT emit additional audit rows — the `voidedContracts` companion is structured data, not an audit log. If you add a fourth `emitAudit` call here, you double-count the void event.
- **Snapshot timing for `priorPaymentTotalCents`:** capture before `internalCancelContract` runs (Task 2's order). Doing it after the helper call is technically fine (helper doesn't touch payments), but the explicit BEFORE-capture is defense-in-depth and makes future refactors safer.
- **Searching for interments via `lots.has_interment` flag:** there is no such denormalized flag. Query `occupants.by_lot` and filter on `dateOfInterment !== undefined`. If the cemetery ever adds the denorm flag (Epic 4 might, for query performance), update this story's call site — but don't add the flag here.
- **Test row-count assertions:** when asserting "no writes occurred," count rows on EVERY table you might have touched — `contracts`, `lots`, `ownerships`, `voidedContracts`, `payments`, `receipts`, `paymentAllocations`, `auditLog`, `receiptCounter`. Missing one creates a false sense of safety.
