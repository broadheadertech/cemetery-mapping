## The workflow this repo is built around

This repo uses the **BMAD Method** (Breakthrough Method of Agile AI-driven Development) — a sequenced agent pipeline: Analyst → PM → UX → Architect → SM → Dev. Each role is a skill prefixed `bmad-agent-*` (e.g. `bmad-agent-pm` = "John", `bmad-agent-architect` = "Winston"). The user may address an agent by name; check `[agents.*]` in [_bmad/config.toml](_bmad/config.toml) for the name→role mapping.

Workflow skills (invoke via the Skill tool when the user asks for them, do not guess):

- **Planning artifacts** land in [_bmad-output/planning-artifacts/](_bmad-output/planning-artifacts/) — produced by `bmad-create-prd`, `bmad-create-ux-design`, `bmad-create-architecture`, `bmad-create-epics-and-stories`, `bmad-sprint-planning`.
- **Implementation artifacts** (story files, dev outputs) land in [_bmad-output/implementation-artifacts/](_bmad-output/implementation-artifacts/) — produced by `bmad-create-story`, `bmad-dev-story`, `bmad-quick-dev`.
- **Project knowledge** (long-lived docs, architecture decisions, project context) lives in [docs/](docs/) — see `bmad-document-project`, `bmad-generate-project-context`, `bmad-index-docs`.

Output folder paths are configured in [_bmad/bmm/config.yaml](_bmad/bmm/config.yaml) — respect them; don't write planning docs into the repo root.

User-level BMAD settings (skill level: `intermediate`, language: English): [_bmad/config.user.toml](_bmad/config.user.toml).

## Architecture intent (from the brief — not yet implemented)

The brief in §7 commits to specific technical decisions that the Architect phase must honor. The most consequential ones for any future code:

- **Stack:** Next.js (App Router, TS) front end + **Convex** as the entire backend (reactive DB, server functions, file storage, scheduled jobs, auth). There is no separate API layer, no ORM, no PostGIS.
- **Schema lives in `convex/schema.ts`** — that file (when it exists) is the canonical data model. Do not propose ERDs or SQL DDL as the source of truth.
- **Lot geometry is a first-class field on every lot document** from day one — lat/lng centroid plus polygon vertices as arrays, with indexed bounding-box fields for viewport queries. The Phase 1 map may render from a static image/SVG, but the schema must already carry geometry so the Phase 2 Leaflet migration is a rendering swap, not a data migration.
- **Viewport-based lot loading**, not "load all 2,000 lots." Reporting queries that need aggregation should be Convex queries with appropriate indexes or pre-aggregated summary docs updated on write — not ad-hoc scans.
- **Role enforcement lives inside Convex functions**, not just in the UI. Every mutation/query must check the caller's role server-side.
- **Atomic multi-document writes** (payment + contract update + receipt generation) belong inside a single Convex mutation. Treat financial transactions as one unit of work.
- **Philippines/BIR receipt compliance** is in scope and non-trivial — see §10 open questions before designing receipt generation.

## Open questions gate

§10 of the brief lists 10 open questions (installment grace/penalty policy, lot types & pricing, BIR receipt requirements, legacy data condition, commission tracking, ownership transfer policy, perpetual care fees, expense categories, expense approval workflow, named user counts). **Phase 1 dev should not start until these are answered.** If the user asks you to design contract logic, receipt formats, expense workflows, or auth seeding before those answers exist, surface the relevant open question rather than guessing.

## Working in this repo

- This is **not a git repository**. Don't run `git` commands expecting state; if the user wants version control, ask before initializing.
- Don't create application directories (`app/`, `convex/`, `src/`) speculatively. Wait for the Architect phase or an explicit user instruction.
- Don't duplicate content from the brief into new docs — link to it. The brief is the source of truth until a PRD supersedes it.
- When a planning skill produces a doc, place it in the configured output folder ([_bmad-output/planning-artifacts/](_bmad-output/planning-artifacts/) or [_bmad-output/implementation-artifacts/](_bmad-output/implementation-artifacts/)) — not in the repo root, not in `docs/` unless it's a long-lived knowledge artifact.
